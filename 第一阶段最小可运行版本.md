# LangCall 第一阶段最小可运行版本技术文档

## 1. 这份文档是给谁看的

这是一份专门写给 `0 基础小白` 的技术文档。

如果你现在的情况是下面这样，这份文档就很适合你：

- 你知道这个项目最终会很复杂，但现在还不想一下子做完全部模块
- 你想先把 `LangGraph` 的主逻辑跑通
- 你还不想马上接入 `Webhook`
- 你还不想马上接入 `PostgreSQL`
- 你还不想马上接入 `Redis`
- 你希望先做一个“最小能跑”的版本，先获得正反馈

这份文档的目标不是让你一次性完成整个大项目，而是先完成 `第一阶段最小可运行版本`。

---

## 2. 第一阶段到底要做什么

第一阶段只做一件事：

`给系统一段假的通话文本 -> 用 LangGraph 跑完整个分析流程 -> 调用大模型 -> 输出结构化 JSON 结果`

先注意，这里有两个关键词：

### 2.1 假的通话文本

因为我们现在还没有真实的通话平台，也还没有接 Webhook，所以先自己手动准备一段测试文本。

### 2.2 结构化 JSON 结果

我们不是要模型随便返回一段话，而是要它返回程序能继续处理的结构化结果。

例如：

```json
{
  "summary": "客户咨询课程价格，希望进一步了解优惠。",
  "sentiment": "neutral",
  "follow_up_needed": true,
  "key_points": [
    "客户咨询课程价格",
    "客户想进一步了解优惠",
    "建议销售继续跟进"
  ],
  "next_action": "24小时内电话回访"
}
```

## 3. 为什么现在不建议一上来就做全部模块

很多初学者最容易犯的错误，就是一开始就想把下面这些全做了：

- Webhook
- FastAPI
- 数据库
- Redis
- LangGraph
- LiteLLM
- 邮件发送
- Docker
- Langfuse

结果通常是：

1. 代码还没开始写，环境就已经把自己搞乱了。
2. 一个地方报错，根本不知道问题出在哪。
3. 项目越做越乱，信心越来越低。

所以更合理的思路是：

先把 `AI 核心主流程` 跑通，再一层层往外补模块。

第一阶段只聚焦这条主线：

1. 输入一段文本
2. 清洗文本
3. 脱敏文本
4. 组装 Prompt
5. 调用模型
6. 校验输出
7. 打印最终结果

只要这 7 步成功，你的项目核心就已经活了。

## 4. 第一阶段完成后，你应该得到什么

当第一阶段完成后，你应该能在终端里运行一条命令：

```bash
python scripts/run_demo.py
```

然后看到类似下面的输出：

1. 原始通话文本
2. 脱敏后的文本
3. 模型返回结果
4. Pydantic 校验后的结构化 JSON

如果你能跑出这些结果，就说明：

- `LangGraph` 流程通了
- 大模型调用通了
- 输出结构校验通了

这时候再往后接 Webhook、数据库、Redis，难度会小很多。

## 7. 每个文件分别负责什么

这是第一阶段最重要的一节。  
你一定要先理解“每个文件干什么”，再去写代码。

### 7.1 `requirements.txt`

作用：保存项目依赖。

第一阶段最小依赖建议如下：

```txt
langgraph
litellm
pydantic
pydantic-settings
python-dotenv
```

这几个依赖分别负责：

- `langgraph`：构建工作流
- `litellm`：调用大模型
- `pydantic`：校验输出结构
- `pydantic-settings`：读取配置
- `python-dotenv`：读取 `.env` 文件

### 7.2 `.env`

作用：保存环境变量，也就是一些不适合直接写进代码的配置。

第一阶段最小配置示例：

```env
LITELLM_MODEL=gpt-4o-mini
OPENAI_API_KEY=your_openai_api_key
```

如果你用的是 Anthropic，也可以写成：

```env
LITELLM_MODEL=claude-3-5-sonnet-latest
ANTHROPIC_API_KEY=your_anthropic_api_key
```

注意：

`.env` 里的 `your_openai_api_key` 只是占位符，你要替换成你自己的真实密钥。

### 7.3 `app/core/config.py`

作用：统一读取环境变量。

你可以把它理解成：  
项目里凡是要用到模型名、API Key 的地方，都不要自己乱写，而是统一从这里读取。

这个文件至少要负责读取：

1. `LITELLM_MODEL`
2. `OPENAI_API_KEY`
3. `ANTHROPIC_API_KEY`

### 7.4 `app/schemas/llm_output.py`

作用：定义模型输出结果的标准结构。

这是整个项目非常关键的一步。

因为我们不希望模型返回“看起来像答案，但程序没法继续处理”的一段自然语言。  
我们希望它返回的是有固定字段的结构化结果。

第一阶段建议定义这些字段：

1. `summary`
2. `sentiment`
3. `follow_up_needed`
4. `key_points`
5. `next_action`

### 7.5 `app/graph/state.py`

作用：定义 LangGraph 工作流中流转的状态。

你可以把状态理解成一个“任务背包”：

- 当前处理到哪一步
- 当前文本是什么
- 脱敏后的文本是什么
- Prompt 是什么
- 模型返回了什么
- 最终结构化结果是什么

第一阶段最小状态建议包含：

1. `transcript`
2. `masked_transcript`
3. `prompt`
4. `llm_raw_output`
5. `parsed_output`

### 7.6 `app/services/pii_masker.py`

作用：对敏感信息做简单脱敏。

第一阶段不需要做得很复杂，先只处理最常见的两类：

1. 手机号
2. 邮箱

例如：

- `13800138000` 变成 `138****8000`
- `test@example.com` 变成 `t***@example.com`

### 7.7 `app/services/litellm_client.py`

作用：封装一次模型调用。

这个文件只做一件事：

输入 Prompt，返回模型输出文本。

第一阶段先不要在里面做太多事情，比如：

- 不要先做重试
- 不要先做日志系统
- 不要先做 Langfuse
- 不要先做多模型兜底

先只保证“能调通一次”。

### 7.8 `app/graph/nodes.py`

作用：编写 LangGraph 的每一个节点函数。

建议第一阶段只保留 5 个节点：

1. `normalize_input`
2. `mask_pii`
3. `build_prompt`
4. `call_llm`
5. `validate_output`

这 5 个节点分别负责：

#### `normalize_input`

把输入文本做最基础清洗，比如去掉多余空格、处理换行。

#### `mask_pii`

对文本中的手机号、邮箱做脱敏。

#### `build_prompt`

把脱敏后的文本和输出要求拼成一个完整 Prompt。

#### `call_llm`

调用模型，并拿回模型原始输出。

#### `validate_output`

用 `Pydantic` 把模型输出解析成标准结构。

### 7.9 `app/graph/workflow.py`

作用：把上面那 5 个节点真正串起来。

顺序应该是：

```text
normalize_input -> mask_pii -> build_prompt -> call_llm -> validate_output
```

这个文件的本质工作，就是让 `LangGraph` 知道整个处理流程该怎么走。

### 7.10 `scripts/run_demo.py`

作用：运行整个第一阶段演示。

这个文件会做三件事：

1. 手动准备一段测试通话文本
2. 启动工作流
3. 打印最终结果

你可以把它理解成“第一阶段的启动按钮”。

### 7.11 `app/main.py`

作用：这个阶段它不是重点。

第一阶段里，这个文件可以先非常简单，甚至可以暂时不参与主流程。  
保留它只是为了让项目结构以后更容易扩展到 FastAPI。

### 7.12 `README.md`

作用：告诉别人这个最小版本怎么运行。

至少写清楚：

1. 先安装依赖
2. 配置 `.env`
3. 运行 `python scripts/run_demo.py`

## 8. 第一阶段的 LangGraph 工作流长什么样

你可以把第一阶段工作流理解成一个非常小的流程图：

```text
输入通话文本
   ↓
基础清洗
   ↓
PII 脱敏
   ↓
拼接 Prompt
   ↓
调用大模型
   ↓
校验 JSON 结构
   ↓
输出最终结果
```

这条链虽然短，但非常重要。

因为后面所有更复杂的能力，比如：

- Webhook
- 数据库
- Redis
- Langfuse
- 邮件日报

本质上都是围绕这条主链往外扩展。

换句话说：

如果主链都没跑通，后面接再多模块也没意义。

## 9. 第一阶段的数据流动过程

很多初学者看不懂“状态流转”。  
这里用最简单的话说明一次。

假设你输入的测试文本是：

```text
你好，我叫张三，我的手机号是13800138000，想咨询一下课程价格。
```

那么程序大致会这样处理：

### 第一步：原始输入

程序先拿到这段原始文本。

### 第二步：基础清洗

如果文本里有多余空格、多个换行，先做简单整理。

### 第三步：PII 脱敏

把手机号处理成：

```text
你好，我叫张三，我的手机号是138****8000，想咨询一下课程价格。
```

### 第四步：组装 Prompt

把文本和要求拼接成一段完整提示词，例如：

- 你是一个客服通话分析助手
- 请根据文本生成结构化结果
- 必须返回 JSON
- 字段包括 `summary`、`sentiment`、`follow_up_needed`、`key_points`、`next_action`

### 第五步：调用大模型

把上一步的 Prompt 发送给模型。

### 第六步：校验输出

如果模型返回的内容符合预期结构，就解析成功。  
如果结构不对，就报错。

### 第七步：打印最终结果

最后把标准 JSON 打印到终端里。

## 10. 为什么第一阶段就要做 Pydantic 校验

很多人会想：

“我都已经让模型返回结果了，为什么还要做校验？”

原因很简单：

模型不是普通函数，它可能会：

1. 漏字段
2. 字段类型写错
3. 把布尔值写成字符串
4. 在 JSON 外面再包一段解释文字

如果你不校验，后面系统就会越来越不稳定。

所以第一阶段虽然很小，但必须养成正确习惯：

- 模型负责生成内容
- `Pydantic` 负责检查内容是否合法

这是后面项目能做大的基础。

## 13. 第一阶段最容易遇到的错误

### 13.1 API Key 没配置

现象：

- 程序调用模型时报错
- 提示鉴权失败

排查方式：

检查 `.env` 是否真的写了正确的 Key。

### 13.2 模型名写错

现象：

- 调用时报模型不存在

排查方式：

检查 `LITELLM_MODEL` 是否写对。

### 13.3 忘记激活虚拟环境

现象：

- `ModuleNotFoundError`

排查方式：

先激活 `.venv` 再运行脚本。

### 13.4 模型输出不是标准 JSON

现象：

- Pydantic 解析失败
- JSON 解码失败

排查方式：

检查 Prompt 是否明确要求：

1. 只输出 JSON
2. 不要输出解释文字
3. 字段必须完整

### 13.5 脱敏函数没有生效

现象：

- 手机号或邮箱还保留在原文中

排查方式：

检查正则表达式是否写对。

## 15. 第一阶段的核心意义

很多同学会觉得：

“这个版本好像很小，会不会太简单了？”

其实不会。

因为第一阶段完成的不是“一个小玩具”，而是整个项目最核心的 AI 主链路：

1. 输入文本
2. 处理文本
3. 调用模型
4. 校验结果

这个闭环一旦跑通，后面的数据库、Webhook、Redis、本地部署，都是在这个核心闭环外面继续搭积木。

## 16. 总结

对于 `LangCall` 这个项目，第一阶段最小可运行版本的正确目标，不是追求功能多，而是追求：

1. 结构清晰
2. 主链跑通
3. 输出稳定
4. 便于后续扩展

你现在最应该做的，不是同时接入所有工程模块，而是先把这一条主流程做好：

`假输入 -> LangGraph -> 脱敏 -> Prompt -> LLM -> Pydantic 校验 -> 结构化输出`

只要这一步跑通，你的项目就已经迈出了最关键的一步。

后续所有复杂能力，都是在这个基础上逐步增加，而不是推倒重来。


1. PostgreSQL

  - 先把数据库接上。
  - 原因：你最终要把 txt、原始通话、分析结果都放数据库里，数据库是后续所有模块的基础。
  - 这一阶段要完成：
      - 设计 raw_calls、call_analysis 两张最小表
      - 把现在的 raw_call_repository.py 和 analysis_repository.py 从“打印/写本地 JSON”改成“真正写数据库”
      - 保留现在的 txt -> CallRecord -> LangGraph 流程不变
  - 完成标准：
      - 运行 python scripts/run_demo.py
      - 不仅终端有输出，数据库里也有对应记录

  2. TXT 导入数据库

  - 把你现有 5 个 txt 文件批量导入数据库。
  - 原因：你后面不应该长期依赖本地文件做主输入，应该尽快切成“数据库是事实来源”。
  - 这一阶段要完成：
      - 增加一个导入脚本
      - 把 data/raw_calls/*.txt 转成 raw_calls 表数据
  - 完成标准：
      - 5 个 txt 都能进入数据库
      - 每条都有 call_id、原文、手机号、邮箱等字段

  3. FastAPI

  - 然后再接 API，不要反过来。
  - 原因：有了数据库之后，API 才有稳定落点；否则 webhook 收到数据也没地方正规保存。
  - 这一阶段要完成：
      - /health
      - /webhooks/calls
      - 先支持“接收 JSON -> 存 raw_calls/task”
  - 完成标准：
      - Postman 或 curl 能调用接口
      - 接口数据能进数据库

  4. 任务表 + Worker

  - 把“接口接收”和“LangGraph 分析”拆开。
  - 原因：这一步是从 demo 走向真实系统的关键。API 不应该直接长时间卡着等模型返回。
  - 这一阶段要完成：
      - 新增 call_tasks 表
      - API 收到请求后只创建任务
      - Worker 单独取任务并执行 LangGraph
  - 完成标准：
      - API 快速返回
      - Worker 能异步处理任务并写分析结果

  5. Redis 幂等 + 分布式锁

  - 不要太早加，在有 API 和 Worker 之后再加最合适。
  - 原因：只有出现重复请求、并发消费时，Redis 的价值才真正体现出来。
  - 这一阶段要完成：
      - 按 event_id / call_id 做幂等
      - 为同一通话加锁，避免重复分析
  - 完成标准：
      - 同一个请求重复发多次，不会重复调用模型和重复写结果
